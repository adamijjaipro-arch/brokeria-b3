04 / 17 Auth Methods

3 Méthodes d'Authentification

📧

Email + Mot de Passe

Identifiants → OTP Email → PIN
  • bcrypt 12 rounds sur le mot de passe
  • OTP 6 chiffres · Redis TTL 10 min
  • Lockout après 3 échecs
✨

Magic Link

Email seul → Token Redis → PIN
  • Lien unique · Redis TTL 15 min
  • Token à usage unique (delete on use)
  • Guard React 18 strict mode (useRef)
🐙

GitHub OAuth

OAuth 2.0 → Callback → PIN
  • Passport.js passport-github2
  • Création compte automatique si inconnu
  • Même chaîne MFA que les autres méthodes

Fix React 18 : const called = useRef(false) empêche le double-appel du useEffect en mode strict.

07 / 17 TOTP

TOTP — RFC 6238 (Time-based OTP)

  • Secret généré avec otplib (authenticator.generateSecret)
  • Stocké chiffré AES-256-GCM + IV aléatoire par chiffrement
  • Fenêtre de validation : ±1 période (tolérance horloge)
  • QR Code généré avec qrcode pour enrollment (Authenticator app)
  • Désenregistrement : supprime secret chiffré de la DB

Sécurité clé : même si la DB est compromise, le secret TOTP ne peut pas être lu sans la clé AES (TOTP_ENCRYPTION_KEY env var, 64 hex chars).

// Chiffrement du secret (AES-256-GCM) const iv = crypto.randomBytes(12) const cipher = createCipheriv( 'aes-256-gcm', key, iv ) encrypted = Buffer.concat([ cipher.update(secret), cipher.final() ]) authTag = cipher.getAuthTag() // Stocké en DB: iv:authTag:encrypted // → impossible à déchiffrer sans TOTP_ENCRYPTION_KEY // Vérification (window ±1) authenticator.verify({ token: dto.code, secret: decryptedSecret, window: 1 })
08 / 17 WebAuthn / FIDO2

WebAuthn / FIDO2 — Biométrie Hardware

  • Library : @simplewebauthn/server v10
  • userVerification: 'required' → force biométrie ou PIN device
  • Challenge unique stocké Redis TTL 300 s
  • Transports supportés : USB, NFC, BLE, internal
  • Credential stocké en DB avec counter (anti-replay)
  • RP ID & Origin configurés via variables d'environnement
Registration → Challenge Redis → Verify & Store → Credential DB
// Options d'enregistrement generateRegistrationOptions({ rpName: 'BrokerIA', rpID: 'localhost', userID: userId, authenticatorSelection: { userVerification: 'required', // force biométrie/PIN device }, excludeCredentials: existingCreds }) // Vérification (API v10) verifyRegistrationResponse({ response: dto, expectedChallenge, expectedOrigin, expectedRPID, }) // registrationInfo.credential.id // registrationInfo.credential.publicKey
09 / 17 Sessions & JWT

Sessions & Tokens JWT

Access Token

  • Durée : 15 minutes
  • Stockage : mémoire Zustand (jamais localStorage)
  • Contenu : userId, email, JTI, iat, exp

Refresh Token

  • Durée : 7 jours
  • Stockage : httpOnly cookie (inaccessible JS)
  • JTI stocké Redis → révocation immédiate possible

Rotation JTI

  • Chaque /auth/refresh invalide l'ancien JTI
  • Token volé ne peut plus être réutilisé
// Clés Redis pour les sessions 'refresh:{jti}' → refresh token 'preauth:{token}' → pré-auth step 1 'pinauth:{token}' → pré-auth step 2 'magic:{token}' → magic link 'otp:{userId}' → OTP email 'fail:pass:{userId}' → compteur échecs 'locked:{userId}' → verrouillage 'fail:ip:{ip}' → échecs par IP // Pas de tokens en localStorage // Access token → Zustand (mémoire) // Refresh token → httpOnly cookie
10 / 17 Sécurité

Sécurité Applicative

  • Brute-force : lockout après MAX_AUTH_FAILURES=3 tentatives
  • Durée lockout : LOCK_TTL_SECONDS=1800 (30 min, configurable)
  • Suivi IP : ≥ MAX_IP_FAILURES=10 → événement SUSPICIOUS_IP
  • CORS strict : configuré via CORS_ORIGINS env var
  • Validation DTOs : class-validator sur tous les inputs
  • Hachage PIN : bcrypt 10 rounds, jamais stocké en clair
3 échecs → ACCOUNT_LOCKED → Redis TTL 30min → SIEM log
// Variables configurables (.env) MAX_AUTH_FAILURES=3 LOCK_TTL_SECONDS=1800 MAX_IP_FAILURES=10 // Suivi IP (trackIpFailure) await redis.incr(`fail:ip:${ip}`) if (count >= maxIpFailures) { await logging.authFailure( null, 'SUSPICIOUS_IP', ip ) } // CORS (main.ts) app.enableCors({ origin: process.env.CORS_ORIGINS, credentials: true, })
